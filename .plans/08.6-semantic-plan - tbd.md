## 1. The load-bearing decision — semantics are CONFIG-DRIVEN, not per-language C++

**DSS Code Prime's thesis is that the source language is 100% configurable.** Lexing and parsing are already fully schema-driven (`.lang.json` → `GrammarSchema`). Semantic analysis MUST be too. A factory that dispatches on `schema.name()` to hand-written `ToySemantics` / `CSubsetSemantics` / `TsqlSemantics` C++ classes would **hardcode** semantics per language and betray the thesis — explicitly rejected.

**Therefore:**

- A new **`semantics` block in the `.lang.json`** (additive — schema **v4**) declares the per-language semantic structure: which rules are declarations and the roles of their children (name / type / initializer / params / body), which rules/tokens are identifier references, which rules open a lexical scope, the built-in type-name → core-`TypeKind` map, and literal-kind → `TypeKind` typing.
- A **single, language-agnostic `SchemaDrivenSemantics`** reads that block and drives symbol/scope/type analysis for **any** language. **Zero per-language C++.**
- **Onboarding a language = adding a `semantics` block to its config.** SE2 (c-subset) and SE3 (tsql) are therefore **pure config + tests** — that is the proof of the universal-compiler thesis, not three more C++ files.
- A `LanguageSemantics` interface is retained as the seam the analyzer talks to (and a clean test boundary / escape hatch for a future language whose needs outgrow the config vocabulary), but its **only shipped implementation in v1 is `SchemaDrivenSemantics`**. If c-subset/tsql reveal a missing descriptor, we extend the **config vocabulary** (additive, like the v2 schema-expressiveness rounds did for operator tables / lexer modes / string styles), never add language-branching C++.

This mirrors the established pattern exactly: v2 made grammar more expressive via config; v4 makes *semantics* expressive via config.

## 2. The `semantics` schema block (v4, additive)

Sketch (toy); fields are additive and optional — absence ⇒ that semantic facet is simply not analyzed for the language.

```jsonc
"semantics": {
  // A rule that introduces a named declaration, and where its parts live as
  // VISIBLE-child indices (EmptySpace skipped — same indexing the Views use).
  "declarations": [
    { "rule": "varDecl",      "name": 1, "init": 3 },
    { "rule": "functionDecl", "name": 0, "params": 1, "body": 2, "kind": "function" }
  ],
  // Node shapes that are identifier USES (resolve against the scope chain).
  "references": [ { "rule": "expression" } ],
  // Rules whose subtree opens a fresh lexical scope.
  "scopes": [ "functionDecl", "block" ],
  // Source type-name → core TypeKind (for type-annotation resolution).
  "builtinTypes": [ { "name": "int", "core": "I64" }, { "name": "bool", "core": "Bool" } ],
  // Literal token-kind → core TypeKind (for literal expression typing).
  "literalTypes": [ { "literal": "IntLiteral", "core": "I64" }, { "literal": "BoolLiteral", "core": "Bool" } ]
}
```

Loader emits `C_*` config diagnostics on malformed/dangling references (unknown rule, out-of-range child index, unknown `TypeKind` name): reuse `C_MissingField`/`C_UnknownShape`/`C_UnknownToken` + new `C_InvalidSemantics` as needed. Bumps `kMaxSchemaVersion` 3 → 4; v1/v2/v3 configs without a `semantics` block stay valid (the facet is just inert).

> **Genericity guard (tripwire):** the analyzer must contain **no** `if (schemaName == "...")`. SE1 adds a test asserting the toy results are produced purely from config, and the engine is exercised against a second synthetic schema to prove no toy-specific assumptions leaked.

## 3. Architecture

```
analyze(shared_ptr<CompilationUnit const> cu) -> SemanticModel
   │  (shared_ptr keeps the CU at a stable address — UnitAttribute holds raw Tree*)
   ├─ TypeLattice{cu->id(), cu->schema().name()} + registerSchemaTypeExtensions
   ├─ SchemaDrivenSemantics sem{cu->schema()}      // reads the v4 `semantics` block
   ├─ Pass 1 (pre-order, per tree): declarations + scope push/pop + redecl check
   ├─ Pass 2 (post-order, per tree): resolve uses + infer literal/expr types + basic checks
   ├─ crossRefs(): inject imported declarations into the importing tree's scope
   └─ SemanticModel{ cu, lattice, scopes, symbols, nodeToSymbol, nodeToType, diagnostics }
```

**Two passes** give forward-reference resolution (G-209) for free: all declarations are minted in Pass 1 before any use is resolved in Pass 2. Cross-tree symbols are CU-wide (the scope tree spans the CU); `crossRefs()` (from CU4) gate which foreign declarations a tree can see.

**`SemanticModel`** (move-only) is the phase output and the LSP/HIR contract:
- `symbolAt(NodeId) -> SymbolId`, `recordFor(SymbolId) -> SymbolRecord const*`
- `typeAt(NodeId) -> TypeId`
- `lattice()`, `scopes()`, `diagnostics()`, `hasErrors()`

**Type rules** (`type_rules.hpp`, header-only) supply what the lattice deliberately omits: `isAssignable` / `isArithmetic` / `unify` over `TypeInterner const&`. SE1 ships the minimal set toy's basic checks need; SE2/SE3 extend (integer ranks, pointers, nullable).

**`S_*` diagnostics:** new band, high-nibble `0xE000` → printed letter `S` (extend `diagnosticCodePrefix`): `S_UndeclaredIdentifier` `0xE001`, `S_RedeclaredSymbol` `0xE002`, `S_TypeMismatch` `0xE003`, `S_NotCallable` `0xE004`, `S_ArgCountMismatch` `0xE005`, `S_ConstViolation` `0xE006`, `S_UnknownType` `0xE007` (extension-type misuse reuses the existing `C_UnknownTypeExtension`/`C_TypeExtensionParamMismatch`).

## 5. SE1 test plan (STRICT asserts)
- Single decl `var x = ...;` → one symbol; `nodeToSymbol` bound at the name node.
- Forward reference across decls → resolves (no `S_UndeclaredIdentifier`).
- Undeclared use → exactly one `S_UndeclaredIdentifier` on the right span.
- Redeclaration in same scope → `S_RedeclaredSymbol` with a `RelatedLocation` to the first decl.
- Nested-scope shadowing → two distinct `SymbolId`s, no redecl error.
- Multi-tree CU: symbol declared in tree A, used in tree B via a crossRef → resolves.
- Basic type check: incompatible assignment / non-arithmetic operands → `S_TypeMismatch` (exact code + span).
- Empty CU → no diagnostics, valid model.
- `static_assert` `SemanticModel` is move-only; death test: `symbolAt` on a foreign-CU NodeId aborts (the `UnitAttribute` cross-CU guard).
- **Genericity:** drive the analyzer over a second hand-built schema (different rule names/child positions) and assert correct symbols/types — proving nothing is toy-specific.

## 6. Risks
- **Config-vocabulary completeness.** The `semantics` block may not capture every c-subset/tsql nuance in SE1. Mitigation: design the vocabulary against all three grammars up front (done in §2 sketch); extend additively in SE2/SE3 exactly as v2 extended the grammar config — never by language-branching the engine.
- **Child-index brittleness.** Visible-child indices couple config to grammar shape. Acceptable for v1 (the Views already use this), and a `C_InvalidSemantics` load check catches out-of-range indices. A future round can add match-by-child-kind roles if needed.
- **CU lifetime.** `SemanticModel` must not outlive / must own the CU stably — solved via `shared_ptr<CompilationUnit const>` so the `UnitAttribute` raw `Tree*` stay valid.
- **Error-recovery trees.** Declaration name nodes may be `Missing`/`Synthetic`; guard with `IdentifierView::from` before minting (as `populateDeclarationSymbols` already does); suppress cascade `S_TypeMismatch` when either operand type is `Invalid`.
