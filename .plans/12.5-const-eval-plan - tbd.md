# Constants-Evaluation Engine — Sub-Plan (12.5)

> A small substrate-tier sub-PR slotted **immediately after the MIR-globals + Cast substrate sequence** ([`12-mir-lir-plan`](./12-mir-lir-plan%20-%20tbd.md) §ML2-substrate). Lifts the inline constant-folding logic that ML2's globals lowering grew (`tryConstFold` in `src/mir/lowering/hir_to_mir.cpp`) into a shared, language-blind **constants-evaluation engine** in `src/core/types/type_lattice/`. Every consumer that needs to compute a compile-time value — HIR verifier's array-length / enum-value rules, MIR-globals' fold-first init policy, the v1 optimizer's constant-fold pass, and D5.5's enum-discriminant evaluation — goes through one source of truth.

> **Why now, not folded into ML2 / D5.** The fold-first MIR-globals cycle proved the surface area (Literal + Unary + integer Binary, with div-by-zero and out-of-range-shift refusals). D5.5 enum values cannot land cleanly without it (each enum case is a constant expression). Float-init folding for MIR globals is currently deferred pending this engine. Three consumers, one shared shape — landing it as a substrate sub-PR rather than a per-consumer bolt-on is the standing "no gaps, no follow-ups, no workarounds" decision.

## 1. Motivation

ML2's globals lowering needs compile-time evaluation of initializer expressions (`int g = 1 + 2;` should fold to literal 3, not synthesize a runtime init function). The inline `tryConstFold` in `hir_to_mir.cpp` handles `Literal` + integer `UnaryOp(Neg/BitNot)` + integer `BinaryOp` (Add/Sub/Mul/Div/Rem/BitAnd/Or/Xor/Shl/Shr + 6 comparisons) on integer literals — enough for v1 c-subset's corpus.

But:

- **Float literal arithmetic** is deferred because float semantics differ (signaling NaN, rounding mode, IEEE 754 exception flags) — a one-line `static_cast<double>` would be incorrect.
- **D5.5 enums** need to evaluate each case's discriminant expression at semantic-phase time (before MIR even exists). A second copy of the integer folding logic in the semantic layer would be the classic "four divergent copies of the same engineering pattern" hazard 08.5's introduction calls out.
- **The HIR verifier** already does narrow constant checks (`S_NonConstantArrayLength` for `int x[N]` — N must be a constant), each implemented per-rule. Sharing those through one engine clarifies the contract: "this is what 'constant' means in this language".
- **Cast-fold** (e.g., `(int8)256 → -128`) needs a target-type-aware truncation/extension policy that the inline ML2 code can't express because it doesn't know the cast operator's full semantics.

Land the engine once. Every caller delegates.

## 2. Design

### 2.1 Files

```
src/hir/
├── const_eval.hpp           # public API (declarations only)
└── const_eval.cpp           # implementation
```

**Layering note.** The engine WALKS HIR (it reads `Hir::kind`/`children`/
`payload` and `HirLiteralPool::at`), so its dependency stack is
`type_lattice + hir`. The earlier draft of this plan placed it under
`src/core/types/type_lattice/`, but the lattice library is BELOW HIR in
the build graph (HIR depends on lattice's `TypeInterner` for type
queries) — putting the engine there would force a cycle. Corrected to
`src/hir/` at CE1 time. The engine returns a `HirLiteralValue` (the same
variant the HIR uses); MIR callers do a trivial field-by-field copy into
a `MirLiteralValue` (the two pools are structurally identical), keeping
the MIR-pool's identity off the lattice/HIR public surface.

### 2.2 Public API

```cpp
namespace dss {

// Successful folds return a `HirLiteralValue` directly — same variant the
// HIR uses for source literals (declared in `hir/lowering/hir_literal_pool.hpp`),
// no separate pool type to bridge. The earlier draft of this plan named a
// distinct `ConstValue` struct; CE1 collapsed it into HirLiteralValue
// because the two are structurally identical and the indirection added
// no value.

// Outcome of a fold attempt. `value` set on success; `failure` describes
// the blocking reason on partial-or-no-fold (so the caller decides whether
// to surface a diagnostic, fall back to a runtime path, or wait for more
// type information).
enum class ConstEvalFailure : std::uint8_t {
    None,                     // success
    NotAConstantExpression,   // operand references a non-constant symbol
    DivisionByZero,
    ShiftCountOutOfRange,
    Overflow,                 // policy-controlled; see EvalOptions
    UnsupportedOperator,      // op not modelled by this engine (e.g. Call)
    UnsupportedTypeKind,      // e.g. float folding when allowFloat==false
};

struct ConstEvalResult {
    std::optional<ConstValue> value;
    ConstEvalFailure          failure = ConstEvalFailure::None;
    HirNodeId                 blamedNode{};  // for diagnostic anchoring
};

// Caller-facing knobs. Different consumers want different policies:
//   - MIR-globals: refuse-to-fold on overflow (defers to runtime). 
//   - D5.5 enums:  hard-fail on overflow (`S_EnumValueOverflow`).
//   - HIR verifier: hard-fail on division-by-zero too (refuses the
//     program rather than the fold).
struct EvalOptions {
    bool allowFloat   = false;   // open in a later cycle; off by v1 default
    bool refuseOnOverflow      = true;
    bool refuseOnDivByZero     = true;
    bool refuseOnShiftOutOfRange = true;
};

// The engine. Takes the HIR (because subexpressions are HIR nodes), the
// interner (for type queries + promotion rules), and the literal pool
// (where Literal nodes' decoded values live). Walks the expression tree
// bottom-up. Pure function — no diagnostics, no mutation of HIR.
[[nodiscard]] ConstEvalResult
evaluateConstant(Hir const& hir,
                 HirLiteralPool const& literals,
                 TypeInterner const& interner,
                 HirNodeId expr,
                 EvalOptions options = {});

} // namespace dss
```

### 2.3 Folding surface

| HIR construct | Fold? | Notes |
|---|---|---|
| `Literal` | ✓ | Direct map from `HirLiteralValue`. |
| `UnaryOp(Neg/BitNot/Plus)` | ✓ | Apply to folded operand. Float-Neg gated on `allowFloat`. |
| `BinaryOp` arithmetic | ✓ | int (Add/Sub/Mul/Div/Rem). Div/Rem-by-0 → `DivisionByZero`. |
| `BinaryOp` bitwise | ✓ | int (BitAnd/Or/Xor). |
| `BinaryOp` shift | ✓ | int (Shl/Shr). Out-of-range count → `ShiftCountOutOfRange`. |
| `BinaryOp` comparison | ✓ | All 6 (Eq/Ne/Lt/Le/Gt/Ge) on int operands. Float gated. |
| `LogicalAnd` / `LogicalOr` | ✓ | Short-circuit. |
| `Ternary` | ✓ | Evaluate cond, then the chosen arm. |
| `Cast` | ✓ | Target-type-aware truncate/extend (closes the Cast-fold gap). |
| `Ref` to a CONSTANT-bound symbol | ✓ | E.g. another enum case in the same enum, or a constexpr `const int g = 7;`. Symbol resolution is the caller's responsibility — passed in as a small `ConstSymbolTable` callback so the engine stays language-blind. |
| `Ref` to a non-constant symbol | ✗ | `NotAConstantExpression`, blamed at the Ref. |
| `MemberAccess` / `Index` | ✗ (v1) | Aggregate constant evaluation deferred. |
| `Call` / `IntrinsicCall` | ✗ (v1) | `constexpr` function evaluation is post-v1. |
| `SizeOf` | ✗ (real blocker) | Needs the TypeInterner layout API which doesn't exist yet (tracked separately as the SizeOf substrate). |
| `AddressOf` / `Deref` | ✗ | Pointers aren't constants in this engine's scope. |
| Float arithmetic | ✗ unless `allowFloat` | Opens in a follow-up cycle that adds the IEEE 754 policy table. |

### 2.4 Promotion rules

Reuses `TypeInterner::commonType` (added by the Cast-emission cycle in the substrate-PR). Integer promotions: `i8`/`u8`/`bool`/`char` → `i32` before any binary op; `int + long → long`; same-width signed-vs-unsigned ties go to unsigned. Mirrors C99 UAC, applied uniformly across all callers.

### 2.5 Diagnostic vocabulary

The engine itself emits NO diagnostics — it returns `ConstEvalResult` and the caller decides:

- **MIR-globals (`hir_to_mir.cpp`)**: on `NotAConstantExpression` or any other failure → silently fall back to the runtime-init function. The engine is an *optimization* here, not a correctness gate.
- **D5.5 enums (semantic phase)**: on any failure → emit `S_EnumValueNotConstant` / `S_EnumValueOverflow` / `S_EnumValueDivByZero` at the blamed node. Refuses the program.
- **HIR verifier (array-length, etc.)**: existing `S_NonConstantArrayLength` re-routes through the engine; richer failures (e.g. negative length) become new verifier codes named in plan 09's deferred-rule list.

## 4. Open questions

| # | Question | Default if unanswered |
|---|----------|-----------------------|
| 1 | Should the engine cache results per `HirNodeId`? Same constant expression evaluated multiple times across phases (MIR-globals + verifier + optimizer) recomputes today. | **No cache in v1.** A per-CU `HirAttribute<ConstValue>` cache can be added when profiling shows it matters; premature caching adds invalidation discipline for no measured win. |
| 2 | Negative array length: `int a[-1];` — engine returns the negative value successfully; verifier emits the diagnostic. Or engine refuses to fold negative-for-array contexts? | **Engine returns success; the caller's diagnostic emits if its context forbids it.** Keeps the engine pure — context-policy stays at the caller. |
| 3 | Overflow policy on integer promotion (`(int32)0x80000000 - 1` — wraps or refuses)? | **Wrap by default** (C99 UB on signed overflow, but predictable wrap is what every real compiler emits when the value is then USED rather than reasoned about). `refuseOnOverflow` knob available to callers that want stricter checking (HIR verifier may flip it on for array lengths). |
| 4 | `Cast` to a type-extension (`(MyVec3)…`) — does the engine know how to fold? | **No — `UnsupportedTypeKind`.** Extension type semantics live in the per-extension lowering registry, not in the engine. Cast to a core lattice type always folds. |

## 6. Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| The engine's policy choices (overflow, rounding) diverge from what HIR / MIR / optimizer expect | Medium | High | `EvalOptions` knobs are caller-controlled; defaults documented; every caller picks its own. |
| Engine becomes a dumping ground for "any compile-time computation" | Medium | Medium | Scope is explicit (table in §2.3); `SizeOf`, aggregates, `constexpr` calls are explicitly out-of-scope. |
| Performance: bottom-up recursive walk over deep expressions | Low | Low | v1 corpus has shallow constant exprs (≤4 levels deep). Caching is question #1's escape hatch when profiling demands it. |
