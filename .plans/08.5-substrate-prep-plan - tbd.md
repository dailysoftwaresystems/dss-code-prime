# Substrate Prep — Sub-Plan (08.5)

> Two-PR refactor landing between [`08-compilation-unit-plan`](./08-compilation-unit-plan%20-%20tbd.md) and the semantic phase (master plan #8). Generalizes the engineering primitives that `Tree` proved out (arena + strong IDs + immutable-post-build + `NodeAttribute<T>` side-tables) into a **reusable substrate** so [HIR](./09-hir-plan%20-%20tbd.md), [MIR and LIR](./12-mir-lir-plan%20-%20tbd.md) can all be built on the same plumbing rather than copy-pasting it. Also lands the **core type lattice + per-language extension registry** that semantic, HIR, FFI and transpile all consume.
>
> **Why now, not later.** The semantic phase wants to populate `NodeAttribute<TypeId>`. The HIR phase wants its own arena. The MIR phase wants its own arena. If we let each phase invent its own arena code, we'll have four divergent copies of the same engineering pattern within a quarter. Land the generalization once before semantic locks in.

## 1. Motivation

`Tree` is built on a clean pattern: a `detail::Node`-keyed dense arena, monotonic strong IDs with cross-arena `treeTag` guard, immutable post-`finish()`, and `NodeAttribute<T>` side-tables that auto-promote sparse↔dense. The pattern is right. The problem is that **the implementation is hardcoded to `Tree` / `Node` / `NodeId` / `TreeId`** — `NodeAttribute<T>` literally takes a `Tree const&`.

The next four layers (HIR, MIR, LIR, plus the symbol table inside `CompilationUnit`) all want the same shape:
- An owned dense arena of POD nodes.
- Strong IDs stamped with an arena tag for cross-arena safety.
- Immutable handoff to consumers.
- Side-tables that decorate without mutating.

If each phase invents its own, we get four divergent move-semantics bugs, four cross-arena guards with different fatal messages, and four duplicate auto-promotion heuristics. SH3's hard-won cross-tree discipline rots quietly across four implementations.

The second concern is the **type system**. Master plan §9 and the [user's three decisions](./00-compiler-implementation-plan%20-%20tbd.md) commit us to a core lattice + per-language extensions. Semantic phase #8 cannot start until that lattice exists — every `NodeAttribute<TypeId>` write needs to point at *something*.

Both belong before semantic, both touch substrate, both are best landed together so the review burden is one substrate-tier pass rather than two.

## 2. Design

### 2.1 SP1 — Generalized arena substrate

A new header-only template family in `src/core/substrate/`:

```cpp
namespace dss::substrate {

// One template parametric over the node POD type and the strong-ID type.
// Tree becomes ArenaContainer<detail::Node, NodeId, TreeId> with the
// existing semantics; HIR / MIR / LIR each instantiate their own.
template <class NodePod, class IdType, class TagType>
class ArenaContainer { ... };

// Generalized attribute side-table; previously NodeAttribute<T>, now
// parametric over the arena it binds to. Same dual storage + auto-promote.
template <class ArenaT, class T>
class ArenaAttribute { ... };

} // namespace dss::substrate
```

Then in `src/core/types/`:

```cpp
using Tree = substrate::ArenaContainer<detail::Node, NodeId, TreeId>;
template <class T> using NodeAttribute = substrate::ArenaAttribute<Tree, T>;
```

- **Zero behavior change** for existing `Tree` consumers — type aliases preserve the public API.
- **All existing tests continue to pass.** The refactor is internal.
- The new `ArenaContainer` carries the cross-arena `tag` discipline (SH3 template); `ArenaAttribute<ArenaT, T>` validates `tag` on every access, same fatal-message style.
- HIR (plan 09), MIR/LIR (plan 12), and the symbol table (phase #8) instantiate `ArenaContainer` with their own POD shapes.

### 2.2 SP2 — Core type lattice + extension registry

A new sibling under `src/core/types/`:

```
src/core/types/type_lattice/
├── core_type.hpp           # the universal lattice
├── type_id.hpp             # DSS_STRONG_ID(TypeId, TypeKindId)
├── type_interner.hpp       # canonicalizes identical types to single TypeId
├── type_registry.hpp       # per-language extension registry
└── type_lattice.cpp
```

**Core lattice members** (universal, hardcoded):

| Family | Members |
|---|---|
| Primitives | `Bool`, `I8`/`I16`/`I32`/`I64`/`I128`, `U8`..`U128`, `F16`/`F32`/`F64`/`F128`, `Char` (Unicode codepoint), `Byte`, `Void` |
| Aggregates | `Struct{name, fields[]}`, `Union{name, variants[]}`, `Tuple{elements[]}`, `Array<T, N>`, `Slice<T>` (ptr+len) |
| SIMD | `Vector<T, N>`, `Matrix<T, R, C>` — useful for non-shader SIMD code (C++ intrinsics, autovec); shader-stage codegen consumes them too |
| Indirection | `Ptr<T>`, `Ref<T>`, `FnPtr<sig>`, `Nullable<T>`, `Optional<T>` |
| Functions | `FnSig{params[], result, cc}` where `cc` is one of `CcSysV`, `CcMS64`, `CcAAPCS64`, `CcApple`, `CcFastcall`, `CcThiscall`, `CcVectorcall`, `CcWasm`, `CcSpirv` |
| Parameterized | `Param<name>` (placeholder), `Bind<base, args[]>` |

`TypeKind` is an **open enum** (extensible) per the registry below. Core members use the reserved range `[0..255]`; extensions get `>= 256`.

**Stage-specific resource types are extensions, not core.** Shader-only types — `Sampler`, `Texture<dim, format>`, `UAV<T>`, `ConstantBuffer<T>`, `WorkgroupShared<T>` — register as extensions via the schema's `typeExtensions[]` field (see below). Same applies to HDL-only types (`Std_Logic`, `Signal<T>`, `Clock`) when [`19-hir-hw-reserved-plan`](./19-hir-hw-reserved-plan%20-%20tbd.md) opens. This keeps the core lattice narrow + reusable; non-shader languages don't pay for shader types they don't use.

**Per-language extension registry**:

```cpp
class DSS_EXPORT TypeRegistry {
public:
    // A language schema (or FFI importer) calls this at load time.
    TypeKindId registerExtension(std::string_view name,
                                 ExtensionDescriptor descriptor);

    // Lookup at type-checking time.
    [[nodiscard]] std::optional<TypeKindId> findExtension(std::string_view name) const noexcept;

    // Iterate (for transpile-map validation).
    [[nodiscard]] std::span<ExtensionDescriptor const> extensions() const noexcept;
};

struct ExtensionDescriptor {
    std::string name;                       // e.g. "TSQL::Varchar"
    TypeKindId kindId;                      // monotonic, >= 256
    std::vector<TypeParam> parameters;      // e.g. {"N": Integer}
    std::string sourceLanguage;             // schema that owns it
};
```

- **Lattice is CU-scoped** (lives on `CompilationUnit`, per CU3 contract). One CU → one interner → one `TypeRegistry`. Cross-CU types go via FFI extern-decl ingestion at link time.
- **Schema integration**: `.lang.json` files gain an optional `typeExtensions[]` array (additive — `dssSchemaVersion >= 3`); each entry adds an extension descriptor at language-load time. v2 configs (no extensions) work unchanged.
- **Diagnostic codes**: `C_UnknownTypeExtension`, `C_TypeExtensionParamMismatch`, plus the existing `C_*` for malformed schemas.

## 4. Open questions

| # | Question | Default if unanswered |
|---|----------|-----------------------|
| 1 | Do we make `Tree` itself `ArenaContainer<...>` (typedef) or keep `Tree` as a thin wrapper with extra non-arena state (schema handle, diagnostics, etc.)? | **Thin wrapper.** `Tree` retains schema + diagnostics + BufferId; only the node-arena slice moves under `ArenaContainer`. HIR/MIR don't carry those, so the substrate primitive stays clean. |
| 2 | Does `ArenaContainer` enforce immutability post-`finish()` at the type level (move-only build → frozen container)? | **Yes.** Mirror `Tree`'s existing `TreeBuilder`-to-`Tree` handoff via `finish() &&`. Builder type vs frozen type at compile time. |
| 3 | Is `TypeRegistry` global, per-CU, or per-tree? | **Per-CU.** Symbol table is already CU-scoped (CU3); the lattice it indexes shares the scope. |
| 4 | Are extension types canonicalized across languages? `C++::Boxed<int>` and `C#::Boxed<int>` — same `TypeId` or different? | **Different.** Extensions are nominal; language-name-qualified. Transpile maps cross the boundary explicitly. Saves a class of "looks the same, behaves differently" bugs. |
| 5 | Schema-version bump (v2 → v3) just for `typeExtensions[]`? | **Yes** — but additive only; v2 configs with no extensions remain valid. Per the artifact-profile-plan §2.1 precedent. |
| 6 | Where do **calling-convention** kinds live? | In the core lattice as `CcSysV`, `CcMS64`, etc. — they're machine-shaped, not language-shaped. FFI plan (11) consumes them when ingesting C headers. |

## 6. Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| SP1 refactor breaks the existing `Tree` API in subtle ways (move ops, attribute iteration order) | Medium | High | All existing tests must pass without modification — that's the acceptance gate. Any signature change requires an explicit `[deviation]` note in the §0 status row. |
| `TypeRegistry` design forecloses future shader / HDL extension shapes | Medium | High | Plan 17 (shader-gpu) and reserved plan 19 (HIR-HW for VHDL/Verilog) review SP2 before SP2 ships. Extension descriptors must support parameterized shape (`Texture<dim, format>`) without re-litigating the registry. |
| Move-only `ArenaContainer<Builder>` → `ArenaContainer<Frozen>` typed handoff trips dependent code | Low | Medium | Mirror `Tree` / `TreeBuilder`'s existing pattern verbatim; that pattern is already battle-tested. |
| Lattice interner cost dominates compile time | Low | Low | Profiled in SP2 acceptance; canonical hash is FNV-1a over the kind+params packed buffer; comparable to schema-id interning. |
