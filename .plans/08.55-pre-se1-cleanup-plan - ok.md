## 1. Why now

Thesis decision #4 states the engine has **no per-language C++**. The audit found that's not yet end-to-end true. Surviving violations:

1. **Parser Pratt walker** (`parser.cpp:1420-1422`) hardcodes the wrapper-rule names `binaryExpr` / `unaryExpr` / `postfixExpr` and aborts fatally if absent ŌĆö engine-baked toy naming convention.
2. **Loader** (`grammar_schema_json.cpp:1034-1036`) hardcodes the same three names to skip shape-existence validation ŌĆö same violation, twice.
3. **Tokenizer numeric scanner** (`tokenizer.cpp:190-338`) hand-codes C-style numeric grammar (hex/binary/octal prefixes, exponent letters, suffix chars) ŌĆö paradigm-baked, can't accommodate languages with different numeric syntax.
4. **Pre-interned token kinds** include `BoolLiteral` / `CharLiteral` / `NullLiteral` ŌĆö paradigm assumptions (not every language has bool/char/null literals as syntactic categories).
5. **Toy-shaped headers** `well_known_names.hpp` + `tree_views.hpp` are consumed by *engine* code (parser, loader, the CU3 placeholder), not just tests.
6. **`symbol_population.{hpp,cpp}`** ŌĆö a CU3 placeholder that hardcodes `FunctionDeclView` + `VarDeclView` over toy. Zero production callers; pure test fixture masquerading as engine code under `DSS_EXPORT`.
7. **LSP shipped-language list** (`schema_cache.cpp:14-18`) is a hardcoded `["toy","c-subset","tsql-subset"]` array.
8. **Config gaps** ŌĆö toy stuck at `dssSchemaVersion: 1` (can't carry v2+ facets); tsql-subset has no `typeExtensions[]` block for `VARCHAR(N)`; c-subset/tsql will need to declare token kinds (bool/null) that get demoted from built-ins.

These all land **in one PR** so SE1 can build on a clean foundation.

## 2. Design decisions (locked)

### 2.1 Wrapper-rule names ŌåÆ config

The `expr` shape gains an explicit `wrapperRules` block:

```jsonc
"expression": {
  "expr": {
    "atom":            "atomExpr",
    "minPrecedence":   0,
    "wrapperRules": {
      "binary":  "binaryExpr",     // each language picks its own names
      "unary":   "unaryExpr",
      "postfix": "postfixExpr"
    }
  }
}
```

- **Required when `expr` is declared.** Loader emits `C_MissingField` if `wrapperRules` is absent or any of `binary`/`unary`/`postfix` is empty. NO defaults ŌĆö the language picks the names.
- Loader auto-interns the declared names (replaces the current "auto-intern these three by hardcoded name" mechanism). Loader's existing shape-existence skip-list (`grammar_schema_json.cpp:1034-1036`) reads from the schema's declared wrapper RuleIds, not from `well_known_names`.
- Parser's Pratt walker reads `schema.exprWrapperRules(exprRule)` ŌåÆ `{binary, unary, postfix}` RuleIds. The fatal in `parser.cpp:1435-1438` becomes unreachable (the loader has already validated they exist).
- Diagnostic: `C_MissingField` on missing `wrapperRules`; `C_UnknownShape` is not raised since the names are synthesized rules.

### 2.2 Numeric lexing ŌåÆ `numberStyle` schema block

Each language declares its numeric syntax:

```jsonc
"numberStyle": {
  "decimal":         true,             // permit bare decimal digits
  "integerPrefixes": [                 // optional; empty = decimal only
    { "prefix": "0x", "radix": 16, "digits": "0-9a-fA-F" },
    { "prefix": "0b", "radix": 2,  "digits": "01" },
    { "prefix": "0o", "radix": 8,  "digits": "0-7" }
  ],
  "exponent": {                        // optional; absent = integer only
    "letters":      ["e", "E"],
    "signOptional": true
  },
  "fractionPoint":   ".",              // optional; absent = no float literal
  "digitSeparator":  "_",              // optional; absent = no separators
  "integerSuffixes": ["u","U","l","L","ll","LL","ul","UL","ull","ULL"],
  "floatSuffixes":   ["f","F","d","D"],
  "emitKind": {                        // which token kinds the scanner emits
    "integer": "IntLiteral",
    "float":   "FloatLiteral"
  }
}
```

- **Required** when the schema's `tokens` block doesn't already handle numerics character-by-character (i.e. when the language has multi-character numeric tokens). Loader emits `C_MissingNumberStyle` if a schema declares `IntLiteral`/`FloatLiteral` token kinds in its grammar but no `numberStyle` block.
- Tokenizer's `scanNumber()` is rewritten to walk the active language's `numberStyle` instead of hardcoding C semantics. Invalid suffixes/missing-digits emit `P_MalformedNumber`.
- Toy: minimal `numberStyle` (or absent if toy doesn't lex numbers). C-subset: full C-style. T-SQL: T-SQL style (no binary/octal prefixes; different exponent rules).
- The old `v2-gap-catalog row 14` debt is resolved by this PR.

### 2.3 Pre-interned token kinds ŌåÆ universal categories only

The loader's `kBuiltinTokenKindNames` is reduced to **universal** categories (every plausible programming language can map onto these if it has them; their absence in a config means the language doesn't use that category):

| Kept (universal) | Removed (paradigm-specific) |
|---|---|
| `Identifier` | `BoolLiteral`  |
| `IntLiteral` | `CharLiteral`  |
| `FloatLiteral` | `NullLiteral`  |
| `StringLiteral` | |
| `Eof`, `Error` | |
| `Whitespace`, `Newline` | |

- Languages that have boolean/char/null literal categories declare them in their own `tokens` block (or `keywords` block for `true`/`false`/`null`-as-keyword).
- `LiteralView` is being retired anyway (┬¦2.4), so the consumer is gone.
- C-subset / T-SQL adjust their configs to declare what they actually use.

### 2.4 Retire `well_known_names.hpp` + `tree_views.hpp`

After ┬¦2.1 and the symbol_population deletion (┬¦2.5), neither has engine consumers. Both files are deleted in full. Tests that used them are updated to:
- Direct rule lookups via the schema (`tree.rule(id) == schema.rules().find("myRule")`).
- Direct visible-child indexing where positional access is genuinely intended (a tiny test-only helper `nthVisibleChild(tree, parent, n)` may be extracted to `tests/core/tree_helpers.hpp` if reuse is high).

No "well-known" framework names survive. Every name a language uses comes from that language's config.

### 2.5 Delete `symbol_population.{hpp,cpp}` + its tests

Zero production callers; hardcoded toy view names. SE1's `SchemaDrivenSemantics` is the real mechanism. Delete now (per "no deferring without a real blocker"), don't wait for SE1.

### 2.6 LSP shipped-language list ŌåÆ directory scan

`src/lsp/schema_cache.cpp:14-18`'s hardcoded array becomes a directory scan over the shipped-config directory at startup (filtering on `*.lang.json`). `--schema-dir` users already bypass this list; the result is that even the LSP's pre-warm path has zero hardcoded language names.

## 3. Schema version

This is all additive within **schema v4** (same version `imports` introduced). `kMaxSchemaVersion` stays at 4. New optional fields (`wrapperRules`, `numberStyle`) are required only when their gate is met (`expr` shape declared ŌåÆ `wrapperRules` required; `IntLiteral`/`FloatLiteral` declared as multi-char tokens ŌåÆ `numberStyle` required).

Configs must be at `dssSchemaVersion >= 4` to use these. Toy gets bumped to 4 as part of this PR.

## 5. New diagnostic codes

- `C_MissingWrapperRules = 0xC02D` ŌĆö `expr` shape declared without complete `wrapperRules`.
- `C_MissingNumberStyle = 0xC02E` ŌĆö language declares numeric literal tokens but no `numberStyle`.

Both in the 0xC band; render as `C02D` / `C02E`.

## 6. Test plan

- All existing tests must pass. Tests that referenced retired headers/views are updated to direct schema/Tree lookups.
- New strict tests for each design facet: missing `wrapperRules` ŌåÆ `C_MissingWrapperRules`; missing `numberStyle` ŌåÆ `C_MissingNumberStyle`; numeric-style fixtures across all three shipped languages (each language's `numberStyle` exercised on representative input).
- Genericity tests: synthesize a non-shipped schema (e.g. wrapperRules with names `"bExpr"`/`"uExpr"`/`"pExpr"`) and verify the Pratt walker drives correctly. Synthesize a non-C numberStyle (e.g. `0$ff` hex, `^` exponent, separator `'`) and verify scanNumber accepts it.
- Death tests: a config that uses `expr` but omits `wrapperRules` must fail load, not abort at parse time. A config that declares `IntLiteral` without `numberStyle` must fail load.

## 7. Risks

- **`scanNumber()` rewrite** is the largest single change. Risk: regressions in numeric lexing across the c-subset corpus. Mitigation: preserve every existing `tokenizer` / `parser_c_subset_smoke` test as-is, only configs change; if a test fails, the new scanner is wrong, not the test.
- **Test breakage from retired views** is broad. Mitigation: update tests file-by-file; each test should make its intent explicit (direct rule lookup) rather than relying on a global toy-shaped helper.
- **Schema-version policy** stays at v4 to keep the surface stable. If a downstream PR (SE1, HR1) genuinely needs v5, that's a separate decision then.
